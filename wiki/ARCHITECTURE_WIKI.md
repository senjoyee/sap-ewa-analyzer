## 1. What is EWA Analyzer?

**EWA Analyzer** processes SAP **Early Watch Alert (EWA)** reports — periodic system health assessments generated by SAP for their customers — and applies generative AI to turn them into actionable intelligence.

### What an EWA Report Contains

An EWA is delivered as a ZIP archive containing an HTML file (converted from SAP's PDF). It is structured as a numbered chapter document (e.g., Chapter 1: Executive Summary, Chapter 11: Security, Chapter 18: Database). Each chapter may flag issues in three colours: **RED** (critical), **YELLOW/AMBER** (warning), or **GREEN** (acceptable).

### What the Application Does

1. **Ingest** — User uploads a ZIP containing the EWA HTML. System extracts HTML → converts to Markdown → stores in Azure Blob Storage.
2. **Classify** — An AI dispatcher reads each chapter title and content snippet and routes it to one of 6 domain specialists.
3. **Analyse** — Six specialist AI agents run in parallel, each extracting RED/YELLOW findings from the chapters assigned to their domain.
4. **Synthesise** — A "Deep Thinker" agent reads all 6 specialist outputs and derives cross-domain implicit risks (e.g., database growth rate implying a future performance crisis).
5. **Export** — All findings + synthesised insights are written to a professionally formatted Excel workbook with an Executive Summary tab and up to 6 domain tabs.
6. **Review** — Users view a summary dashboard in the web UI, download the Excel, and can chat with an AI about the report via an interactive chat window.

### The 6 Domains

| Domain | ID Prefix | Focus |
|--------|-----------|-------|
| Security | SEC | Auth, patches, vulnerability posture |
| Database | DB | HANA / DB2 / Oracle — growth, capacity, fragmentation |
| Performance | PERF | CPU, memory, dialog response times, work processes |
| Basis / Technical | BAS | Kernel, transport system, system parameters |
| Business | BIZ | Business processes (ECC/S4 systems only) |
| Lifecycle Management | LCM | Software version, maintenance windows, end-of-life dates |

## 4. End-to-End Data Flow

### 4.1 Upload Flow

```
User selects ZIP file + enters metadata (customer, SID, date range)
    ↓
POST /api/upload  (FormData: file, customer_name, system_id, start_date, end_date)
    ↓
Backend:
  1. Validates ZIP, extracts HTML file from archive
  2. Converts HTML → Markdown (html_markdown_converter.py)
  3. Generates canonical blob name: {SID}_{DD}_{MON}_{YY}.md  (e.g. ERP_07_Jun_25.md)
  4. Uploads .md to Azure Blob Storage with metadata tags:
     customer_name, system_id, report_date, start_date
    ↓
Response: { filename, original_filename, customer_name, system_id, report_date, message }
```

### 4.2 Analysis Pipeline (the core)

```
User clicks "Process" on a file in the UI
    ↓
POST /api/reprocess-ai  { blob_name: "ERP_07_Jun_25.md" }
    ↓
Backend (ai_router.py → workflow_orchestrator.py):

  [Step 1] Set processing=true metadata on blob

  [Step 2] Delete any existing analysis artifacts
           (.json, .xlsx, _workbook.xlsx, _workbook_payload.json, etc.)

  [Step 3] Download markdown from Azure Blob

  [Step 4] execute_workflow() — _run_v2_pipeline():

    [4a] SLICE
         ewa_slicer.slice_chapters(markdown)
         → dict { 0: Preamble, 1: Chapter1, ..., N: ChapterN }
         Each entry: ChapterData { number, title, raw_content, subsections[] }

    [4b] DISPATCH (LLM routing)
         For every chapter (except preamble), call LLM in parallel:
           Input:  chapter title + first 1200 chars of content
           Prompt: router_system_prompt.md
           Output: { "domain": "security|database|...", "confidence": 0.0-1.0 }
         Confidence < 0.70 → fall back to static lookup table
         Result: domain_chapters { security: [ch11], database: [ch18,ch19], ... }

    [4c] SPECIALIST AGENTS (6× parallel via asyncio.gather)
         Each agent receives its list of ChapterData objects.
         Concatenates chapters → calls LLM with domain-specific prompt.
         LLM produces JSON with:
           findings[]   — RED/YELLOW findings with recommendations
           parameters[] — SAP parameter change suggestions
           abstentions[] — chapters with no findings (coverage proof)
         Findings IDs normalised: SEC-01, DB-02, PERF-03, BAS-04, BIZ-05, LCM-06

    [4d] DEEP THINKER (after all specialists complete)
         Input:  all 6 DomainResult objects (serialised to JSON)
         Prompt: deep_thinker_prompt.md
         Output: supplemental_findings[] (max 8), IDs: DT-01 … DT-08
         Looks for: growth trajectories, cross-domain anti-patterns,
                    abstention clusters that imply coverage gaps

    [4e] BUILD WORKBOOK
         excel_workbook_builder.build_workbook(domain_results, supplemental_findings)
         Executive Summary tab + up to 6 domain tabs (see section 5.9)

  [Step 5] Upload artifacts to Azure Blob:
     {base}_workbook.xlsx
     {base}_workbook_payload.json   (all domain_results + supplemental_findings)
     {base}_routing_map.json        (per-chapter routing decisions + confidence)
     {base}_v2_usage.json           (token usage per step)

  [Step 6] Clear processing flag, set last_status = "completed" | "failed"
    ↓
Response: { success, workbook_file, workbook_payload_file, total_findings, total_parameters, supplemental_findings }
```

### 4.3 File Status Detection

The frontend polls `GET /api/files` every 10 seconds. The backend inspects all blobs in the container and derives status from the artifact set:

| Artifacts present | Derived Status |
|-------------------|---------------|
| Only `.md` | "New" (uploaded, not processed) |
| `.md` + `_workbook_payload.json` + `.xlsx` | "Analyzed" |
| metadata `processing=true` | "Processing" |
| metadata `last_status=failed` | "Failed" |

### 4.4 Chat Flow

```
User opens chat popup for an analyzed file
    ↓
chat.html?fileName=ERP_07_Jun_25.md
    ↓
POST /api/chat  {
  message: "What are the top security findings?",
  fileName: "ERP_07_Jun_25.md",
  chatHistory: [{ text, isUser }]
}
    ↓
Backend (chat_router.py):
  1. Fetches {base}.md (full report text) from Blob
  2. Fetches {base}_workbook_payload.json (structured findings, optional)
  3. Builds system prompt injecting: report context + workbook JSON
  4. Trims history to last CHAT_HISTORY_LIMIT (10) messages
  5. Calls Azure OpenAI Responses API (reasoning: "medium")
    ↓
Response: { response: "markdown text" }
    ↓
Frontend renders markdown (tables, bold, code blocks, headings, lists)
```

## 5. Backend Architecture

### 5.1 Application Entry Point

**File**: `backend/ewa_main.py`

- Creates FastAPI app instance
- Reads `CORS_ALLOWED_ORIGINS` env (comma-separated origins) → defaults to localhost 3000/5000/8080
- Initialises Azure BlobServiceClient (singleton)
- Registers `XSUAAMiddleware` (JWT auth — only active on Cloud Foundry)
- Registers CORS middleware (allowed methods: GET, POST, DELETE, OPTIONS)
- Includes all 5 routers under `/api` prefix
- Runs Uvicorn on `0.0.0.0:8001`

### 5.2 API Routers & Contracts

#### Storage Router (`/api`)

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/upload` | Upload ZIP → extract HTML → convert to MD → store in Blob |
| GET | `/files` | List all files with derived status, metadata, and grouping |
| DELETE | `/delete-analysis` | Delete a file and all its derived artifacts |
| GET | `/download/{blob_name}` | Stream a blob file with appropriate content-type |

**POST /api/upload**
- Request: `multipart/form-data`
  - `file`: ZIP file
  - `customer_name`: string
  - `system_id`: string (SAP SID, typically 3 chars)
  - `start_date`: `YYYY-MM-DD`
  - `end_date`: `YYYY-MM-DD`
- Blob naming: `{SID}_{DD}_{MON}_{YY}.md`
- Stored metadata tags on blob: `customer_name`, `system_id`, `report_date`, `start_date`
- Response: `{ filename, original_filename, customer_name, system_id, report_date, message }`

**GET /api/files**
- Response: `{ files: [...], sequential_groups: [...] }`
- Each file object:
  ```json
  {
    "name": "ERP_07_Jun_25.md",
    "last_modified": "2025-06-07T12:00:00Z",
    "size": 245000,
    "report_date": "2025-06-07",
    "report_date_str": "07.06.2025",
    "customer_name": "Acme Corp",
    "system_id": "ERP",
    "processed": true,
    "ai_analyzed": true,
    "processing": false,
    "last_status": "completed",
    "last_error_hint": null,
    "last_error_message": null,
    "sequential_processing_available": false,
    "files_in_group": 1,
    "unprocessed_in_group": 0
  }
  ```
- Logic: scans all blobs, checks for artifact set, reads metadata, groups by customer+SID

**DELETE /api/delete-analysis**
- Request: `{ fileName: string, baseName: string }`
- Deletes: `.md`, `.json`, `.xlsx`, `_AI.json`, `_AI.md`, `_workbook.xlsx`, `_workbook_payload.json`, `_v2_usage.json`, `_parameters.json`, `_token_usage.json`, `_routing_map.json`
- Security: rejects `..`, `/`, `\` in names (path traversal guard)

**GET /api/download/{blob_name}**
- Streams blob content with MIME type based on extension
- Content-Disposition: attachment for xlsx, inline for others
- 404 if not found

#### AI Router (`/api`)

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/process-and-analyze` | Full analysis pipeline |
| POST | `/analyze-ai` | Re-run AI on existing markdown |
| POST | `/reprocess-ai` | Same as process-and-analyze (frontend calls this) |

All three accept: `{ blob_name: string }`

All three respond with:
```json
{
  "success": true,
  "message": "Workflow completed",
  "workbook_file": "ERP_07_Jun_25_workbook.xlsx",
  "workbook_payload_file": "ERP_07_Jun_25_workbook_payload.json",
  "usage_file": "ERP_07_Jun_25_v2_usage.json",
  "total_findings": 42,
  "total_parameters": 18,
  "supplemental_findings": 5
}
```

Blob metadata during/after processing:
- `processing`: `"true"` | `"false"`
- `last_status`: `"completed"` | `"failed"` | `"processing"`
- `last_error_hint`: short description on failure
- `last_error_message`: full error text on failure

#### Chat Router (`/api`)

**POST /api/chat**
- Request:
  ```json
  {
    "message": "string",
    "fileName": "ERP_07_Jun_25.md",
    "documentContent": "optional override text",
    "fileOrigin": "optional hint",
    "contentLength": 0,
    "chatHistory": [{ "text": "string", "isUser": true }]
  }
  ```
- Response: `{ "response": "markdown string" }`
- Fetches markdown + workbook payload from Blob
- Injects workbook findings JSON into system prompt for structured context
- Last `CHAT_HISTORY_LIMIT` (default 10) messages from history included

#### Export Router (`/api`)

**GET /api/export-excel?blob_name={name}**
- Normalises blob_name to find `{base}_workbook.xlsx`
- If pre-generated: streams it directly
- If not found: falls back to regenerating from `_workbook_payload.json` via `build_workbook()`
- Response: Binary XLSX, filename `{base}_{customer}.xlsx`

#### Health Router (`/api`)

**GET /api/ping**
- Response: `{ "status": "ok", "message": "Server is running" }`
- Auth bypassed (public path)

### 5.3 Workflow Orchestrator

**File**: `backend/workflow_orchestrator.py`

The orchestrator is the brain of the backend. It implements the `EWAWorkflowOrchestrator` class.

**State management**: Uses `WorkflowState` dataclass:
```
WorkflowState {
  blob_name:              str
  markdown_content:       str
  domain_results:         list[DomainResult]
  supplemental_findings:  list[SupplementalFinding]
  workbook_bytes:         bytes
  routing_map:            dict[str, RoutingEntry]
}
```

**Main entry point**: `execute_workflow(blob_name, skip_markdown=False)`

Pipeline steps:
1. **Download** — fetch markdown from Blob Storage
2. **V2 Pipeline** — `_run_v2_pipeline()`:
   - Slice → Dispatch → Specialists → Deep Thinker → Workbook
3. **Save** — upload all artifacts to Blob, write metadata

**AI client initialisation**:
- Reads `PROVIDER` env → selects `AnthropicFoundry` or `AzureOpenAI`
- Different model names for router / specialist / deep thinker roles
- All passed to agents at construction time

### 5.4 Document Ingestion & Conversion

**Files**: `backend/converters/html_markdown_converter.py`, `pdf_markdown_converter.py`

**HTML to Markdown** (primary path for ZIP uploads):
- Extracts HTML from ZIP archive
- Parses with BeautifulSoup4
- Strips navigation, scripts, styles
- Converts heading tags → `#` markdown headings
- Converts table tags → markdown pipe tables
- Converts list tags → `- ` items
- Strips preamble content before "Chapter 1" heading

**PDF to Markdown** (legacy / alternative path):
- Uses PyMuPDF4LLM (or Azure Form Recognizer)
- Less reliable table extraction than HTML path
- Retained for backward compatibility

**Important**: The canonical input format for all downstream processing is **Markdown**. All converters produce the same Markdown structure regardless of source format.

### 5.5 Chapter Slicing

**File**: `backend/utils/ewa_slicer.py`

Deterministic, no LLM required.

**Algorithm**:
```
regex: ^#\s+(\d+)\s+(.*)   →  matches "# 11 Security Audit"
regex: ^##\s+(?:(\d+\.\d+)\s+)?(.*)  →  matches "## 11.1 User Authentication"
```

1. Find all `# N Title` markers in the markdown
2. Slice content between consecutive markers
3. Content before first chapter → `chapter[0]` (preamble, skipped in dispatch)
4. Within each chapter, find `## ...` markers → extract subsections as `(title, content)` tuples

**Output**: `dict[int, ChapterData]`
```python
ChapterData {
  number:      int              # Chapter number (0 = preamble)
  title:       str              # Heading text
  raw_content: str              # Full body text
  subsections: list[(str, str)] # [(subtitle, subcontent)]
}
```

### 5.6 AI Dispatcher (Chapter Router)

**File**: `backend/utils/ewa_dispatcher.py`

**Purpose**: Assigns every chapter to exactly one of the 6 domains.

**Algorithm**:
1. Raises immediately if no AI client provided — there is no fallback path
2. For each chapter (except preamble), build a routing prompt:
   - System prompt: `router_system_prompt.md` (describes domain boundaries)
   - User content: chapter title + first 1200 characters of body
3. Call LLM for ALL chapters simultaneously via `asyncio.gather()`
4. Parse LLM response: `{ "domain": "security" }`
5. If domain not in the 6 known domains → raise `ValueError` → workflow fails
6. Any API error propagates immediately → workflow fails

**Output**:
- `domain_chapters`: `dict[domain_name → list[ChapterData]]`
- `routing_map`: `dict[chapter_num → RoutingEntry]` for diagnostics

**RoutingEntry**:
```python
{
  chapter_number: int
  title:          str
  domain:         str
  source:         "llm"   # always
}
```

### 5.7 Specialist Agents (6 Domains)

**File**: `backend/agent/specialist_agents.py`

**Design**: One base class `SpecialistAgent`, six subclasses, all run in parallel.

**Base class responsibilities**:
1. Load domain prompt from `prompts/specialist_{domain}.md`
2. Load JSON schema from `schemas/specialist_{domain}_schema.json`
3. Concatenate assigned chapters with separators
4. Call LLM (provider-specific) with structured output mode
5. Validate/repair response
6. Normalise finding IDs to domain prefix
7. Ensure coverage: every input chapter appears in findings OR abstentions

**Input** (per agent): `list[ChapterData]` (chapters routed to this domain)

**Output**: `DomainResult`
```python
DomainResult {
  domain:       str                    # "security" | "database" | etc.
  findings:     list[dict]             # RED/YELLOW findings extracted
  parameters:   list[dict]             # SAP parameter change recommendations
  abstentions:  list[dict]             # Chapters with no findings
  applicable:   bool                   # False for business on non-ECC systems
  usage:        dict                   # Token counts
}
```

**Finding structure** (per finding):
```json
{
  "finding_id":      "SEC-01",
  "chapter":         11,
  "title":           "Missing security patches",
  "rag_status":      "RED",
  "finding":         "Description of the issue",
  "recommendation":  "SAP's recommended action",
  "priority":        "High"
}
```

**Parallel execution**:
```python
tasks = [agent.run(domain_chapters[domain]) for agent in [security, database, ...]]
results = await asyncio.gather(*tasks, return_exceptions=True)
```
Failed agents return an empty `DomainResult` (graceful degradation).

**Domain subclasses**: `SecuritySpecialist`, `DatabaseSpecialist`, `PerformanceSpecialist`, `BasisSpecialist`, `BusinessSpecialist`, `LifecycleSpecialist`

### 5.8 Deep Thinker Agent

**File**: `backend/agent/deep_thinker_agent.py`

**Purpose**: Second-pass cross-domain synthesis. Identifies risks that no single domain agent would flag because they span multiple domains.

**Input**: All 6 `DomainResult` objects, serialised to JSON

**Prompt logic** (`prompts/deep_thinker_prompt.md`):
- Look for growth trajectories (e.g., DB adding 50GB/month while performance already degraded)
- Look for anti-patterns (e.g., no patching + no monitoring + HIGH severity alert)
- Look for abstention clusters (multiple domains have no findings in same chapter range → may indicate content wasn't routed correctly)
- Conservative rule: only derive a finding with clear multi-domain evidence
- Maximum 8 supplemental findings

**Output**: `list[SupplementalFinding]`
```python
SupplementalFinding {
  finding_id:     "DT-01"
  title:          str
  domain:         str   # Primary domain affected
  finding:        str   # The derived insight
  rationale:      str   # Why this conclusion was drawn
  recommendation: str   # What to do
  source:         "AI Deep Analysis"
}
```

### 5.9 Excel Workbook Generation

**File**: `backend/utils/excel_workbook_builder.py`

**Entry point**: `build_workbook(domain_results, supplemental_findings, domain_chapters, system_metadata) → bytes`

**Sheet structure**: 1 Executive Summary tab + up to 6 domain tabs

| Tab | Contents |
|-----|----------|
| **Executive Summary** | System metadata, overall risk counts, findings heatmap across all domains |
| **Security** | Findings, Parameters, AI Deep Analysis supplements for security domain |
| **Database** | Findings, Parameters, AI Deep Analysis supplements for database domain |
| **Performance** | Findings, Parameters, AI Deep Analysis supplements for performance domain |
| **Basis** | Findings, Parameters, AI Deep Analysis supplements for basis/technical domain |
| **Business** | Findings, Parameters, AI Deep Analysis supplements — **omitted for non-ECC/S4 systems** |
| **Lifecycle** | Findings, Parameters, AI Deep Analysis supplements for lifecycle domain |

Each domain tab has three clearly separated sections: **Findings** (RED/YELLOW/GREEN), **Parameters** (change recommendations), and **AI Deep Analysis** (DT-xx supplemental findings relevant to that domain).

**Business tab logic**: `_is_business_applicable(dr)` — returns `False` when `DomainResult.applicable` is `False` or abstentions contain `reason = "not_applicable_system_type"`. Tab is silently omitted in that case.

**Styling**:
- SAP colour palette: gold `#F0AB00`, dark `#333333`, implicit/DT purple `#6868AC`
- Severity fill colours: RED→`#CC0000`, YELLOW→`#E07000`, GREEN→`#008000`

## 7. AI / LLM Integration

### 7.1 Provider Abstraction

The system supports two AI providers, selected via the `PROVIDER` environment variable:

| Setting | SDK | Models Used |
|---------|-----|-------------|
| `openai` (default) | `openai` Python SDK (AzureOpenAI) | GPT-4.1 / gpt-5.x via Azure OpenAI |
| `anthropic` | `anthropic` SDK (AnthropicFoundry for Azure AI Foundry) | claude-haiku-4.5, claude-sonnet-4.5/4.6 |

Each agent class (`SpecialistAgent`, `DeepThinkerAgent`) has both a `_call_llm_openai()` and `_call_llm_anthropic()` method and dispatches based on the `provider` parameter set at construction time.

### 7.2 Prompt Files

All prompts are external Markdown files loaded at import/construction time:

| File | Used By | Role |
|------|---------|------|
| `prompts/router_system_prompt.md` | ewa_dispatcher.py | Classify chapters to domains |
| `prompts/specialist_security.md` | SecuritySpecialist | Extract security findings |
| `prompts/specialist_database.md` | DatabaseSpecialist | Extract database findings |
| `prompts/specialist_performance.md` | PerformanceSpecialist | Extract performance findings |
| `prompts/specialist_basis.md` | BasisSpecialist | Extract basis/technical findings |
| `prompts/specialist_business.md` | BusinessSpecialist | Extract business findings |
| `prompts/specialist_lifecycle.md` | LifecycleSpecialist | Extract lifecycle findings |
| `prompts/deep_thinker_prompt.md` | DeepThinkerAgent | Cross-domain synthesis |
| `prompts/chat_system_prompt.md` | chat_router.py | Chat persona / instructions |

**Specialist prompt structure** (all 6 follow same pattern):
1. Domain expertise framing
2. Finding extraction rules (RED = SAP says immediate action; YELLOW = SAP says review)
3. **Coverage rule**: every chapter assigned MUST appear in findings OR parameters OR abstentions
4. Recommendation-first approach: extract what SAP explicitly stated, not infer
5. Output JSON schema description (finding_id, chapter, title, rag_status, finding, recommendation, priority)

### 7.3 Structured Output Strategy

Both providers use **JSON schema-constrained output** to guarantee valid JSON:

**OpenAI (Responses API)**:
```python
client.responses.create(
    model=model,
    input=[...],
    text={ "format": { "type": "json_schema", "name": "...", "schema": strict_schema, "strict": True } },
    reasoning={ "effort": reasoning_effort }
)
```

**Anthropic (Messages API)**:
```python
client.messages.create(
    model=model,
    system=[...],
    messages=[...],
    output_config={ "format": { "type": "json_schema", "schema": strict_schema } }
)
```

**Strict schema preprocessing**:
- Adds `"additionalProperties": false` recursively to all object nodes
- Removes `$schema`, `$id` meta-keywords (not accepted by OpenAI)
- Ensures all properties in every object are listed in `required`

**Fallback chain**:
1. Structured output → success → return parsed JSON
2. Structured output → fail → streaming fallback (collect text delta events)
3. Streaming → success → parse JSON text
4. All attempts fail → return empty `DomainResult` (graceful degradation)

**Anthropic extended thinking**: NOT used with structured output (SDK incompatibility). Thinking only applied in streaming fallback path for Deep Thinker.

**Prompt caching** (Anthropic): `cache_control: { "type": "ephemeral" }` on static blocks (system prompt, schema block, document block) to reduce cost on repeated calls.

### 7.4 Token Accounting

Every LLM call returns usage data:

**OpenAI**: `response.usage` → `{ input_tokens, output_tokens, reasoning_tokens }`

**Anthropic**: `response.usage` → `{ input_tokens, output_tokens, cache_read_input_tokens, cache_creation_input_tokens }` + thinking tokens from streaming events

All usage is aggregated in `WorkflowState` and persisted to `{base}_v2_usage.json` with per-step breakdown (dispatch, each specialist, deep thinker).

## 8. Core Infrastructure

### 8.1 Authentication (XSUAA / JWT)

**File**: `backend/core/xsuaa_middleware.py`

**Active only on SAP BTP Cloud Foundry** — detected via CF environment variables. In local dev, all requests pass through.

**Flow**:
1. Check if path is public (bypass list: `/`, `/health`, `/api/ping`, `/docs`, `/openapi.json`, `/redoc`)
2. Extract `Authorization: Bearer <token>` header → 401 if missing
3. Fetch XSUAA public keys from `{VCAP_SERVICES.uaa.url}/token_keys`
4. Cache keys by `kid` (Key ID). On kid-miss → refetch
5. Validate JWT signature (RS256) using matched public key
6. Verify: expiration, issuer matches XSUAA URL, scope contains `{xsappname}.viewer`
7. On success: store `{ id, email, scopes }` in `request.state.user`
8. On failure: 401 (invalid token) or 403 (insufficient scope)

**Roles defined** (in `xs-security.json`):
- `Viewer`: can view and export analyses
- `Administrator`: can also delete analyses

### 8.2 File Storage (Azure Blob)

**File**: `backend/core/azure_clients.py`

Single container, flat blob namespace. Blob names serve as primary keys.

**Naming conventions**:
```

{SID}_{DD}_{MON}_{YY}_workbook_payload.json  ← JSON analysis results
{SID}_{DD}_{MON}_{YY}_routing_map.json    ← dispatcher diagnostics
{SID}_{DD}_{MON}_{YY}_v2_usage.json       ← token usage report
```

**Metadata tags** (stored on the `.md` blob):
- `customer_name`
- `system_id`
- `report_date`
- `start_date`
- `processing` (`"true"` | `"false"`)

**Status derivation** (in `GET /api/files`):
- Enumerate all blobs → group by base name

## 10. Key Design Patterns & Decisions

### Async-First Execution
All LLM calls use `asyncio.to_thread()` to wrap synchronous SDK calls. All 6 specialist agents run concurrently via `asyncio.gather()`. All chapter routing calls also gather in a single round-trip.

### Stateless Backend
No database. All state is in Azure Blob Storage as files. Status is inferred from the artifact set + blob metadata. This means:
- Container restarts = zero data loss
- Simple horizontal scaling (any instance can serve any request)
- Easy debugging (download any blob to inspect state)

### Provider Abstraction
Every AI-calling class has duplicate `_call_llm_openai()` / `_call_llm_anthropic()` methods. This is explicit duplication (not abstracted through a common interface) to allow provider-specific optimisations. The `provider` string is threaded through from the orchestrator to every agent.

### Coverage Guarantee
Every input chapter MUST appear in a specialist's output — either as a finding, a parameter recommendation, or an abstention. This is enforced both in the prompt (instructions) and in code (`_fill_missing_abstentions()` post-processes the LLM response). This prevents silent analysis gaps.

### Graceful Degradation
- Failed specialist → empty DomainResult (pipeline continues)
- Streaming fallback if structured output fails
- All LLM attempts fail → empty DomainResult returned (pipeline continues)

### Blob-Centric Artifact Model
All analysis outputs are separate named blobs. The frontend fetches them individually by convention. This means artifacts can be regenerated independently (re-run AI without re-uploading, re-export Excel without re-analysing).

### Prompt Externalisation
All prompts are Markdown files loaded from disk at startup. Prompt improvements require no code changes — only file edits. This is especially valuable for iterating on agent behaviour.

## 11. Known Limitations & Areas to Revisit

### Chat API URL Hardcoding
`chat.html` contains hardcoded URL detection logic (localhost → port 8001, azurewebsites.net → hardcoded backend URL). This breaks in any new deployment environment. Should be replaced with a config endpoint or environment-injected value.

### docker-compose Frontend Reference
`docker-compose.yml` references a `/frontend` directory that appears to have been replaced by the `sapui5/` directory. The compose file is partially outdated.

### No Request Queuing
Long-running analysis jobs (can take 3–5 minutes) are run synchronously in the HTTP request handler. This is viable for low concurrency but breaks under load. A proper job queue (Celery, RQ, Cloud Tasks) with async status polling would be more robust.

### SAPUI5 Analysis View Is Programmatic
`Analysis.controller.js` builds the analysis view by imperatively creating UI5 controls in JavaScript and appending them to a container. This bypasses UI5's declarative XML view system and is harder to maintain. A proper approach would use data binding with XML view templates.

### No Automated Tests
No automated unit or integration test suite exists. Priority targets for any rewrite: the slicer regex, dispatcher confidence threshold logic, workbook tab structure, and the structured output fallback chain.
